# 生产人工验收 Checklist

## 目标

这份清单用于在“兼容式单项目收口”阶段，对生产环境做人工验收。

适用场景：

- 新版本已部署到生产
- 自动化检查已通过
- 需要确认关键用户链路、运营链路、运维链路没有回退

不适用场景：

- schema 破坏性迁移后的数据级验收
- 多租户平台化能力验收

## 执行原则

- 优先在低峰期执行
- 一次只验证一个链路，验证后立即记录结果
- 遇到高风险异常时先停，不要边在线排障边继续放量
- 验收记录建议包含：
  - 时间
  - 版本 commit
  - 验收人
  - 结果
  - 备注 / 异常截图 / 日志位置

## 0. 验收前确认

- 确认当前线上版本号已记录
- 确认 `.env` 与 `docker-compose.yml` 备份已存在
- 确认数据库快照或 SQL 备份已存在
- 确认回滚目标版本已明确
- 确认：
  - `SINGLE_OWNER_MODE=1`
  - `EXPECTED_TENANT_CODE` 已固定
  - `REQUIRE_EXISTING_TENANT=1`
  - `ALLOW_TENANT_CODE_MISMATCH` 为空

## 1. 基础可用性

- `docker compose ps` 正常
- `/health/ready` 返回 `ok: true`
- `/ops/project-check` 返回 `ok: true`
- 当前 project code 与预期一致
- app / worker 启动日志无持续报错刷屏

通过标准：

- 所有容器处于 `Up`
- `database` / `redis` 检查均为 `true`
- `project-check` 命中当前预期项目

## 2. 旧链接兼容性

- 抽查 3 到 5 个旧 `shareCode`
- 验证是否都能正常打开
- 验证打开后的媒体可正常交付
- 验证第一页打开时没有报错或空白

通过标准：

- 旧分享链接仍可打开
- 无明显“内容不存在 / 权限错误 / 副本缺失”异常

## 3. 新上传链路

- 用主账号上传一个新作品
- 补充标题 / 描述
- 确认生成的新分享入口可打开
- 确认 worker 补副本正常推进
- 确认打开后评论 / 点赞按钮正常

通过标准：

- 上传成功
- 元数据可保存
- 新分享入口可打开
- worker 无持续失败

## 4. 设置读写链路

- 修改一项欢迎词设置
- 修改一项广告 / 按钮设置
- 修改一项搜索开放设置
- 修改一项内容保护或隐藏发布者设置
- 退出后重新进入页面确认设置仍存在

通过标准：

- 设置修改成功
- 刷新页面后配置仍正确
- 没有“设置像被重置”现象

## 5. 搜索与标签链路

- 搜索一个已知关键词
- 搜索一个无结果关键词
- 打开热门标签页
- 进入一个标签详情页
- 验证公开访客与项目成员视角是否符合预期

通过标准：

- 搜索结果与权限一致
- 标签页可正常进入
- 不出现异常空白或越权展示

## 6. 历史 / 足迹 / 列表链路

- 打开社区列表
- 打开我的内容 / 我的足迹
- 切换日期或筛选条件
- 验证分页 / 返回路径

通过标准：

- 列表渲染正常
- 筛选条件生效
- 分页与返回无异常

## 7. 推送链路

- 创建一条推送草稿
- 编辑推送内容
- 查看目标人数
- 创建一个定时推送或测试草稿流
- 如环境允许，完成一次小范围发送验证

通过标准：

- 草稿可创建、可编辑、可取消
- 推送调度不报错
- worker 正常处理推送任务

## 8. Worker / 副本链路

- 检查 worker 进程正常
- 检查最近上传作品的副本状态
- 抽查 `/ops/project-check` 后再看 worker 日志
- 确认没有持续性 `FAILED` 堆积

通过标准：

- worker 正常运行
- 副本补齐不持续卡死
- 无明显异常堆积

## 9. 权限边界

- 普通用户无法进入管理链路
- 非 OWNER 不应获得管理能力
- 单人模式下多管理员入口行为符合预期

通过标准：

- 没有越权
- 单人模式约束生效

## 10. 异常观察点

重点关注这些异常信号：

- `project-check` 命中到非预期项目
- 设置页面看起来像“恢复默认”
- 旧 `shareCode` 打不开
- 新上传后长时间停留在 pending
- worker 日志持续刷同类错误
- app / worker 反复重启

如果出现上述任一高风险问题：

- 立即停止继续验收
- 保留日志
- 决定是否执行回滚

## 12. 当前建议

在当前阶段，这份清单优先用于：

- 每次较大兼容式重构后上线验证
- 准备进入 schema 清理设计前的稳定性确认

只有当这份清单连续多轮都稳定通过时，才建议进入下一阶段的 schema 清理设计与演练。
