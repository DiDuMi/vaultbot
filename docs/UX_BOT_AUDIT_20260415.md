# 交互优化与风格统一建议

## 背景

当前项目的主要交互界面不是传统 Web，而是 Telegram Bot 的：

- 文案
- Inline Keyboard 按钮
- Reply Keyboard 底部菜单
- 多步输入流程

因此“交互混乱”和“风格不统一”本质上集中在信息架构、导航规则、按钮命名、状态反馈四个层面。

## 当前主要问题

### 1. 页面像功能堆叠，不像任务流

高频页面里同时放了很多动作，但缺少主次层级。

典型表现：

- `设置` 同时承载运营数据、管理员、分类、搜索开放、排行开放、广告、存储群、推送等能力
- `我的`、`足迹`、`排行`、`列表` 之间有交叉，但角色边界不够清晰
- 用户常常要先理解“系统有哪些功能”，才能知道下一步该点哪里

结果：

- 新用户不容易建立心智模型
- 租户用户会觉得功能很多，但路径不短
- 某些操作完成后，下一步不够明显

### 2. 返回路径不稳定

虽然大量页面都有“返回”按钮，但返回目标并不总是一致。

典型表现：

- 有的返回上一级功能页
- 有的返回帮助页
- 有的直接回首页
- 某些回收、推送、存储群场景又各自拼了“返回某某页”

结果：

- 用户无法形成稳定预期
- 同一个“返回”动作在不同页面语义不同
- 多步操作里容易有“我现在在哪”感

### 3. 同类动作命名不够统一

当前项目里已经有很多不错的按钮命名，但仍然存在一些风格割裂：

- `返回` / `返回设置` / `返回推送` / `返回内容` / `返回列表`
- `刷新` 出现频率高，但有时是必要操作，有时只是兜底
- `更多` / `收起` 的使用逻辑存在，但没有成为统一的“次级信息展开模式”

结果：

- 用户要重新学习每个模块的按钮语义
- 视觉上像是不同阶段、不同作者分别补出来的页面

### 4. 消息文案偏“说明书”，不够“下一步导向”

很多文案解释得很完整，但对用户最关键的信息其实只有三件事：

1. 你现在在什么地方
2. 当前状态是什么
3. 下一步该做什么

现状里一些页面说明较多，任务指令较弱。

例如：

- 设置页说明内容多，但“最常用入口”不够突出
- 推送页信息较全，但“现在应该编辑什么/发送什么”没有强主导
- 欢迎词、广告、自动归类等输入页，说明存在，但缺少更统一的输入模板感

### 5. 角色切换的体验还不够清楚

系统同时服务：

- 普通用户
- 租户成员
- 管理员

虽然权限判断是清楚的，但体验层面还可以更明确：

- 普通用户看到不可用功能时，往往是“被挡住”
- 管理员拥有更多入口，但没有被组织成“日常运营工作台”

结果：

- 普通用户会觉得有些按钮“点进去才知道不能用”
- 管理员会觉得高级功能散落在设置内部

## 建议的统一原则

### 1. 页面结构统一成三段

每个核心页面都尽量保持这个顺序：

1. 标题
2. 状态摘要
3. 下一步动作

建议模板：

```text
标题

当前状态摘要

你可以：
1. 做动作 A
2. 做动作 B
3. 做动作 C
```

这样用户看到消息第一眼就能判断“这里是干什么的”。

### 2. 固定导航语法

建议统一为：

- 第一行优先放：`⬅️ 返回` + `🏠 首页`
- 最后一行优先放：`🔄 刷新`
- 分页统一：`⬅️ 上一页` + `当前页/总页数` + `下一页 ➡️`
- 展开次级动作统一：`⋯ 更多`
- 收起次级动作统一：`⬆️ 收起`

说明：

- “返回”只表达“回上一层功能页”
- “首页”只表达“回到全局起点”
- 不把“刷新”混在主操作里

### 3. 按钮语义分层

建议把按钮固定分成三类：

- 主动作：打开、编辑、发送、添加、开启
- 次动作：预览、刷新、筛选、更多
- 风险动作：删除、回收、移除、关闭、取消推送

表现规则：

- 主动作尽量放前面、同一行
- 风险动作单独一行
- 风险动作前尽量给确认页

### 4. 文案改成“任务引导式”

建议每个页面文案尽量遵循：

- 第一行说明当前页面
- 第二部分说明当前状态
- 最后一句明确告诉用户下一步动作

例如推送页：

- 当前草稿状态
- 是否有文案 / 按钮 / 媒体
- 下一步建议：先编辑内容，再预览，最后发送

### 5. 按角色拆首页入口

建议继续保留当前“普通用户 / 租户用户”差异化菜单，但进一步强化：

- 普通用户：首页突出 `列表 / 搜索 / 我的`
- 租户成员：首页突出 `分享 / 列表 / 推送 / 设置`
- 管理员：在设置页顶部明确放“常用运营入口”

## 推荐优化顺序

### P0：立刻做

这些改动低风险、高收益：

- 修正文案乱码、按钮命名不一致
- 固定返回/首页/刷新位置
- 所有分页按钮统一写法
- 所有空状态统一写法
- 所有权限阻断统一写法

### P1：本轮重点

- 重构 `设置` 页面，把能力分成“内容管理 / 增长运营 / 系统配置”
- 重构 `推送` 页面，把“当前草稿”和“可执行动作”拆清楚
- 重构 `列表/足迹/排行` 的互跳关系，减少重复入口
- 在输入流程页中统一说明模板

### P2：后续增强

- 为管理员增加“运营驾驶舱”式首页
- 为普通用户增加更明显的“发现内容”路径
- 把搜索、标签、列表统一为一个发现体系
- 给高风险动作加更强的前置确认和结果反馈

## 可以直接落地的页面重组建议

### 首页

普通用户建议保留：

- `📚 列表`
- `🔎 搜索`
- `👤 我的`
- `🏆 排行` 作为次入口

租户建议强化：

- `分享`
- `列表`
- `推送`
- `设置`

如果不想新增底部菜单项，至少在租户首页文案里把“分享”和“推送”列为前两项主任务。

### 设置

建议分成三组：

- 内容管理：分类、隐藏发布者、自动归类、内容保护
- 运营工具：推送、欢迎词、广告、排行开放、搜索开放
- 系统配置：管理员、存储群、通知

这样比当前平铺更容易扫读。

### 推送

建议页面文案固定为：

- 当前选中的推送是谁
- 当前状态是什么
- 是否已配置文案 / 媒体 / 按钮
- 下一步推荐动作

推荐按钮顺序：

1. 编辑内容
2. 配置按钮
3. 预览
4. 立即推送 / 定时推送 / 循环推送
5. 删除草稿 / 取消推送

### 列表 / 搜索 / 足迹 / 排行

这几个页面属于“内容发现”链路，建议统一交互骨架：

- 顶部显示当前视图和筛选状态
- 中部列结果
- 底部固定放分页和跳转
- 扩展动作统一放在 `⋯ 更多`

## 这次代码层面的建议

本轮建议优先做这几件事：

1. 抽出统一导航按钮构造函数，避免每个页面手写。
2. 修正明显的文案字符异常。
3. 把“刷新”统一放到底部，减少按钮层级混乱。
4. 后续再抽“阻断提示”“成功提示”“下一步提示”的统一消息模板。

## 结论

这个项目的问题不是功能不够，而是：

- 功能很多，但缺少稳定的任务结构
- 页面都能用，但不像同一套系统
- 用户能完成操作，但不够轻松

最有效的方向不是“大改视觉”，而是先统一：

- 导航规则
- 按钮语言
- 页面结构
- 角色入口

只要这四层统一，Bot 的交互会立刻从“能用但乱”变成“复杂但清楚”。
