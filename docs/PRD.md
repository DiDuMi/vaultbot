# PRD：Telegram 多租户内容存储与检索系统（纯 Telegram 交付）

## 1. 项目概述

### 1.1 背景
Telegram 近期存在频道/群组被误封或连带封禁的情况，导致博主将内容发布在频道/群中的历史媒体与消息不可达，用户触达链路中断，内容资产与用户关系同时受损。

本项目目标是在“媒体文件仅存 Telegram、用户访问仅通过 Telegram”的前提下，通过多租户能力、内容索引与用户行为数据，把“内容交付”和“用户关系”从单一频道/群/单一租户 Bot 中解耦，降低封禁导致的业务损失。

### 1.2 产品定位
- 面向博主/内容团队的 SaaS：一套系统支持多个租户克隆使用
- 面向用户的内容入口：在 Telegram 内完成搜索、浏览、收藏、点赞与历史回看
- 面向运营：基于租户与用户数据形成画像，支持分群触达与内容推荐

### 1.3 约束与边界
- 不使用域名作为用户访问入口（用户不依赖网页链接访问内容）
- 不使用云对象存储/外部文件存储（媒体文件仅存 Telegram）
- 允许存在后端服务与数据库用于存储“租户/用户/内容元数据与行为数据”，不存媒体文件本体
- Bot 运行优先使用 long polling（getUpdates）模式；如启用 webhook，需要域名与 HTTPS 支持并配置安全校验
- 当前仅支持“租户 Bot 单入口”模式；门户 Bot 模式作为后续重构目标

### 1.4 成功指标（可量化）
- 内容可达性：任意单个存储频道/群或单个租户 Bot 失效时，已绑定用户仍可访问 ≥ 99% 历史内容
- 访问转化：用户完成“搜索→打开内容”的转化率、收藏率、复访率可追踪
- 运营效率：租户可在 1 分钟内完成一次分群推送与效果回收（送达/打开/点击）

### 1.4.1 指标口径（验收）
- 内容可达性（Reachability）按“打开请求”统计：用户在租户 Bot 触发一次打开，即产生一次 open 请求
- 统计窗口：按自然日与近 7 天滚动窗口两种口径输出
- 成功判定：租户 Bot 在单次打开请求内（可重试/可切换副本）成功交付内容即计为成功；因权限不足拒绝不计入失败
- 失败归因需分类：Vault Group 不可用、单副本不可用、429 限流、网络异常、Bot 被封/不可用等

## 2. 角色与权限

### 2.1 角色定义（RBAC）
- Owner：租户所有者，拥有全部权限
- Admin：管理租户配置、成员、内容与运营
- Editor：上传/编辑内容、管理合集与可见性
- Support：客服消息接入与回复
- Analyst：只读查看统计与用户画像

### 2.2 用户身份
- 平台用户：通过租户 Bot 与系统交互的 Telegram 用户
- 租户用户：在某租户上下文下有访问/订阅/权限关系的用户（可同一人属于多个租户）

## 3. 核心概念与术语
- Tenant Bot：租户唯一入口 Bot（搜索、历史、收藏、权限校验、客服、推送）
- Portal Bot：平台统一入口 Bot（未来重构目标）
- Vault Node：租户的媒体存储载体（私有频道或群）
- Asset：逻辑内容对象（一次上传对应一个 Asset，可含多个文件/版本）
- Replica：某 Asset 在某个 Vault Node 的一份副本（message_id + chat_id）
- Collection：合集/目录（用于组织与权限继承）
- Event：用户行为事件（浏览、搜索、点赞、收藏、分享、推送点击等）

## 4. 需求总览（MVP → 增强）

### 4.1 必须实现（MVP）
1) 多租户：租户创建/配置/隔离
2) 租户成员协作：邀请管理员、角色权限
3) 上传与存储：批量上传媒体到 Telegram Vault Nodes，生成内容卡片与可分享方式
4) 内容索引：可按标题/标签/文件名/描述检索（受“可检索开关”约束）
5) 用户行为：点赞、收藏、浏览历史、搜索历史
6) 容灾（Telegram 内）：多副本存储、自动换副本；租户 Bot 被封时尽量缩短恢复时间
7) 内容可见性：公开/受限/分级，支持按用户/标签/角色控制

### 4.2 可选增强（后续）
- 客服：用户在租户上下文发起客服，会话可转发给 Support 并可回复回投
- 全员推送：租户可编辑并群发给租户用户（带速率控制与失败回收）
- OCR/语音转文本用于更强检索（仍仅存索引文本，不存媒体文件）
- 推荐排序（基于画像与行为）
- 防搬运/防外泄策略（更细粒度水印、转发限制策略）

## 5. 功能需求（详细）

### 5.1 多租户与租户管理（纯 Telegram 管理台）
- 租户创建：Owner 在租户 Bot 内完成租户初始化，获得租户码（短字符串）
- 租户配置：
  - 租户名称、简介、默认语言
  - 可检索策略（见 5.4）
  - 默认可见性策略（公开/受限/分级）
  - Vault Nodes 管理（添加/移除/健康状态）
- 租户成员：
  - 邀请成员（通过 Telegram 用户 ID 或邀请口令）
  - 角色授予与回收
  - 操作审计（谁在什么时候做了什么）

### 5.2 上传、存储与副本
#### 5.2.1 上传入口
- 租户 Bot 内上传：Editor/Owner 向租户 Bot 发送文件/媒体消息（支持多条连续消息作为一次批量上传）
- 上传会话：Bot 将一次上传视为一个“上传批次”，可设置标题/描述/标签/合集/可见性
- 保存决策：上传批次进入“待保存”状态，用户可选择“完成保存”或“取消保存”

#### 5.2.2 存储策略（纯 Telegram）
- Vault Node 采用“租户级超级群（启用话题）”，每个租户独享 1~N 个 Vault Group
- 每个 Asset 至少写入 R 份副本（建议 2~3），副本分布在不同 Vault Group（避免单群被封全灭）
- 副本定位数据：vaultGroupChatId + messageThreadId + messageId + file_unique_id（如可得）+ 发送时间 + 校验信息（hash/size）

##### 5.2.2.1 Vault Node 设计
- Vault Group：租户级私有超级群（必须启用话题），群内只允许平台运维与租户 Bot 存在，禁止租户/用户加入
- Vault Topic：每租户一个话题，作为该租户内容的隔离命名空间
- 规模与扩容：
  - 每租户默认 1 个 Vault Group，按需增加 Backup/Cold
  - 当单租户写入量/风险需要提升时，新增 Vault Group 并绑定为 Backup
- 冗余策略（强烈建议默认开启）：
  - 同一租户至少绑定 2 个 Vault Group（Primary/Backup），分别有各自的 Topic
  - 读取优先 Primary，失败自动切到 Backup
- 创建职责：
  - Vault Group 由平台为租户创建并托管（降低租户上手成本与运营成本）
  - Vault Topic 由租户 Bot 在 Vault Group 内自动创建
- Bot 权限要求：租户 Bot 为 Vault Group 管理员，具备创建话题、发消息、复制消息等权限
- 节点元数据（后台保存）：
  - vaultGroupId、chatId、用途（primary/backup/cold）、状态（active/degraded/banned）、权重、创建时间
  - tenantId → messageThreadId 映射（每个绑定的 Vault Group 一份）
  - 最近健康检查时间、成功率、连续失败次数、最后失败原因

##### 5.2.2.2 副本写入（复制优先，重传兜底）
- 写入目标：按节点权重挑选 R 个 active Vault Group 写入副本（写入到该租户对应的 messageThreadId）
- 写入方式优先级：
  1) copyMessage：将上传源消息直接复制到 Vault Group 的租户话题（不重新上传，速度快，失败率低）
  2) forwardMessage：必要时转发（保留来源信息，依场景可禁用）
  3) 重新发送文件：当复制/转发不可用时作为兜底（成本高，尽量少用）
- 上传批次完成条件：R 份副本全部成功写入或达到可配置的最小成功数（例如 min=2）
- 后台落库：每个副本写入成功立即记录 Replica（chatId、messageId、file_unique_id、size、hash、createdAt）
- 哈希与直达码：上传批次在“取消保存/完成保存”时生成文件哈希与直达码，作为租户一键复制入口

##### 5.2.2.3 读取与交付（多副本读容灾）
- 用户打开内容时，租户 Bot 先做权限校验，再选择可用副本交付（优先 Primary Vault Group）
- 副本选择策略（从简单到完善）：
  - 优先选择最近成功的副本
  - 同一用户在短时间内尽量命中同一副本（减少抖动）
  - 副本失败时自动切换到下一副本并记录失败原因
- 交付方式优先级：
  1) copyMessage：从 Vault Group 的租户话题复制到用户对话（推荐，避免用户看到 Vault 来源）
  2) forwardMessage：在允许保留来源时使用
  3) 降级为“内容卡片 + 直达码”：当所有副本交付失败时，给出可重试入口并触发修复流程

##### 5.2.2.4 完整性校验（不依赖外部存储）
- 写入时保存：hash、size、mime、file_unique_id（若可得）
- 抽样校验：按计划任务对热点内容做“可复制/可发送”探测，更新节点与副本健康度
- 失败分类：权限不足/消息不可用/节点被封/429 限流/网络异常等，用于后续自动处置

##### 5.2.2.4.1 取消保存与完成保存
- 取消保存：仅保留已写入的副本与最小索引信息，撤销对外可见性与检索入口
- 完成保存：生成内容卡片与直达码，写入可检索字段并对外发布

##### 5.2.2.5 目录与“应急入口消息”
- 为每个租户生成一条“应急入口消息”（文本），包含：
  - 租户 Bot 主/备用户名列表
  - 租户码
  - 用户自助指引（如何在租户 Bot 粘贴直达码、如何切换租户）
- 该消息需同步复制到该租户绑定的每个 Vault Group 的 Topic（等同多副本），并可定期推送给活跃用户作为离线备份

##### 5.2.2.6 Vault Group 运维规范（平台侧）
创建与配置：
- 平台为租户创建 Vault Group（私有超级群），必须开启话题
- 群内仅允许平台运维账号与租户 Bot 存在，禁止租户/用户加入
- 租户 Bot 必须为管理员，并具备创建话题、发消息、复制消息等权限

规模与容量：
- 默认每租户 1 个 Vault Group（Primary），必要时增加 Backup/Cold
- 冗余策略：每租户绑定 Primary/Backup 两个 Vault Group，并分别创建租户 Topic

话题命名与索引：
- Topic 标题建议包含租户标识与短码，便于人工排障与审计
- 后台维护 tenantId → (vaultGroupChatId, messageThreadId) 映射，并支持版本化（便于迁移/回滚）

健康检查与告警：
- 写路径探测：定期在每个 Vault Group 执行最小写入/复制探测，更新成功率与状态
- 读路径探测：对抽样内容执行“可复制到测试对话”探测，识别坏副本与坏节点
- 告警条件：连续失败超过阈值、429 频繁、写入/复制权限丢失、群不可访问

故障处理与迁移：
 - 节点降级：Vault Group 进入 degraded/banned 后立刻停止写入，并从读路径剔除
 - 扩容：新增 Vault Group 并绑定为 Backup/Cold；按需迁移写入
 - 租户迁移：为租户在新 Vault Group 创建新 Topic，并将后续写入切换到新 Topic；历史内容按“补副本”策略逐步补齐

审计与权限：
- Vault Group 相关操作必须记录审计日志（创建 Vault Group、创建 Topic、绑定/解绑、迁移切换、补副本）
- 不允许在 Vault Group 内进行人工发布内容；所有内容写入必须通过 Bot 执行，便于追溯

能力探测矩阵与降级策略：
- 平台必须维护“Vault Group 能力矩阵”，记录该群在当前 Bot 身份下的可用能力：
  - 是否支持创建话题
  - 是否支持发送到指定话题（message_thread_id）
  - 是否支持 copyMessage 到指定话题
  - 是否支持 forwardMessage 到指定话题
- 能力探测必须在以下时机执行并落库：
  - 新增 Vault Group 后的上线验收
  - 定时巡检（例如每小时/每天）
  - 关键失败触发（连续失败超过阈值时立刻复测）
- 写入路由必须按能力矩阵选择策略（从优到劣）：
  1) copyMessage 到指定话题
  2) sendDocument/sendPhoto/sendVideo 等重新发送到指定话题
  3) 降级：切换到其他 Vault Group 的话题写入（Primary→Backup 或更多副本）
- 若检测到“无法写入到指定话题”，必须将该 Vault Group 标记为 degraded 并停止新增绑定，避免使用不可用节点

##### 5.2.2.7 租户独享 Vault Group 方案的问题与对策（深入分析）
该方案的优势是“隔离性强、风险边界清晰”，但也引入了运维规模与成本压力。需要在 PRD 阶段明确可接受的风险边界与对策。

单点风险：
- 问题：单个 Vault Group 被封会影响该租户全部内容
- 对策：
  - 每租户默认绑定 Primary/Backup 两个 Vault Group（跨账号维度尽量分散）
  - R 份副本必须跨 Vault Group 分布，禁止把多个副本写到同一群
  - 热点租户优先增加 Backup 或迁移到新的 Vault Group

平台运维成本与权限风险：
- 问题：平台必须维护大量租户 Vault Group 的管理员权限与稳定性；管理员权限丢失将导致写入/复制失败
- 对策：
  - 为每个 Vault Group 维护“权限快照”与定期探测（权限不足立即告警）
  - 运维账号与 Bot 权限分离，最小化管理员数量与暴露面
  - 所有写入路径都必须可观测（成功率/429/错误码分类），并有自动降级策略

API 能力不确定性与兼容性：
- 问题：需要确认 Bot API 对“复制/转发到指定话题（message_thread_id）”的支持范围；不同方法的参数支持可能存在差异
- 对策：
  - 设计为“能力探测优先”：在每个 Vault Group 上对 copy/forward/send 三种方式做探测并记录可用矩阵
  - 若 copyMessage 无法指定 thread，则退化为在话题中重新发送文件或先复制到群再二次整理（需评估成本）
  - 所有写入都带幂等键，避免因重试导致重复内容难以治理

Topic 生命周期与误操作：
- 问题：话题可能被重命名、关闭、删除，或被设置为不可写；导致 tenantId → threadId 映射失效
- 对策：
  - 映射必须可版本化：每次迁移/重建 Topic 生成新版本并原子切换写入指针
  - 对每个租户 Topic 定期做“存活探测”，失效时自动进入修复队列（新建 Topic、切换写入、补副本）

数据治理与“群内混乱”：
- 问题：即使按话题隔离，群内仍可能出现重复副本、失败重试造成的脏数据，影响排障与迁移
- 对策：
  - 统一内容写入格式：每个 Asset 先写一条“索引头消息”，后跟文件消息，便于定位与批量处理
  - 维护副本状态机（active/bad/evicted），坏副本不再参与读取与补副本源选择
  - 提供“租户级清理/重建”运维指令：重建 Topic、重建索引头、重新补副本

限流与性能：
- 问题：单个 Vault Group 集中写入会更容易触发 429 限流；同时推送、上传、补副本可能竞争配额
- 对策：
  - 发送队列分级：用户交付 > 上传写入 > 补副本 > 健康检查（严格优先级）
  - 全局与 Vault Group 级限流：按 Vault Group 维度做速率控制与退避
  - 热点租户扩容：把写入量大的租户迁移到新的 Vault Group，降低对单群的影响

安全与隔离：
- 问题：虽然租户/用户不进群，但平台侧集中存储意味着“误配权限”可能导致跨租户读取风险
- 对策：
  - 任何读取/交付必须以 tenantId 为主索引，只从该租户映射到的 threadId 列表中选副本
  - 禁止基于 file_unique_id 全局反查副本并交付（避免跨租户数据串用）
  - 对所有运维操作强审计与双人审批（可后续增强）

结论与取舍：
- 该方案适合追求“租户极简接入”的 SaaS 形态，但必须默认启用 Primary/Backup 跨群冗余与完善的健康检查/修复队列
- 若未来封禁风险进一步上升，可将“热点租户独立 Vault Group”作为常态化配置，而不是例外

#### 5.2.3 内容标识与分享方式（不依赖域名）
在不使用域名的前提下，内容分享以“Telegram 内可传播的信息载体”为主：
- 方式 A：分享“内容卡片消息”（租户 Bot 生成，含标题/描述/按钮）
- 方式 B：分享“租户码 + 内容编号”（可复制文本），用户到租户 Bot 输入即可直达
- 方式 C：若 Vault Node 是公开频道且风险可接受，可分享频道消息链接（作为补充，不作为唯一入口）
- 方式 D：生成文件哈希与哈希链接，提供租户一键复制入口

### 5.3 客服（Support）
- 用户在租户上下文点击“联系客服”进入会话
- 用户消息被转发到租户的 Support 接入点：
  - 方案 1：转发给一个“客服群”（管理员/客服在群内回复）
  - 方案 2：转发给指定 Support 成员私聊
- 回复回投：系统维护“转发消息 ↔ 工单 ↔ 用户”映射，Support 的引用回复可精准回投给用户
- 支持能力：会话分配、快捷回复、黑名单、会话关闭、统计（响应时长/满意度可选）

### 5.4 内容可检索开关与搜索
#### 5.4.1 租户级检索策略
租户可配置 search_mode：
- OFF：任何用户都无法通过搜索发现内容（仅能通过直达方式打开）
- ENTITLED_ONLY：仅对“有权限的用户”可被检索（默认推荐）
- PUBLIC：所有用户可检索（适合公开资源库）

#### 5.4.2 内容/合集级覆盖
- Collection.searchable / Asset.searchable 可覆盖租户默认策略
- deny 优先级高于 allow（例如黑名单永远不可见）

#### 5.4.3 搜索范围与排序
可检索字段：
- 标题、描述、标签、文件名、内容类型、上传者（可选）
排序（MVP）：
- 相关度优先，其次热度（点赞/收藏/打开量），再按时间
过滤器（MVP）：
- 类型（图片/视频/GIF/文件）
- 标签
- 时间范围

### 5.5 点赞、收藏、历史
#### 5.5.1 点赞
- 用户可对 Asset 点赞/取消点赞
- 用于热度排序、画像特征与内容质量反馈

#### 5.5.2 收藏
- 用户可收藏 Asset 到“我的收藏”
- 支持按租户查看收藏列表与搜索

#### 5.5.3 浏览历史
- 记录两级事件：
  - impression：内容卡片曝光
  - open：用户实际打开/请求交付内容
- 支持按租户维度查看“最近浏览”
- 支持在历史中搜索（按标题/标签/描述的索引）

### 5.6 全员推送（Broadcast）
- 租户可创建推送任务：
  - 文本/图文/按钮
  - 目标：全部用户 / 某标签分群 / 某行为人群（增强）
- 发送控制：
  - 速率限制与队列
  - 失败回收（用户拉黑/无法发送/429 等）
  - 暂停/继续/撤回（能力视 Telegram 限制而定）
- 效果统计（MVP）：
  - 送达率（可发送成功/失败）
  - 打开/点击（基于按钮回调与内容打开事件）

### 5.7 内容可见性与权限
权限模型（MVP）：
- Public：所有人可打开
- Protected：需要在租户 Bot 绑定租户上下文后可打开
- Restricted：需满足规则才可打开

规则要素（MVP）：
- allow_roles / allow_tags / allow_user_ids
- deny_user_ids（最高优先级）
继承策略：
- Collection 可设置默认可见性与规则，Asset 可覆盖

交付策略：
- 一律由租户 Bot 在“用户请求打开”时校验权限，再选择可用 Replica 交付

## 6. 容灾与降损（纯 Telegram 可实现的上限）

### 6.1 目标定义
- 单个 Vault Group 被封：系统自动换到其他 Vault Group 副本，用户无感
- Tenant Bot 被封：尽量缩短恢复时间，存量用户通过备用入口继续使用

### 6.2 关键策略
1) 存量用户入口统一在租户 Bot（绑定关系、搜索、历史、收藏都在租户 Bot）
2) 租户 Bot 主/备：预置多套租户 Bot（主/备）共享同一后端与数据库，用户可按应急入口切换
3) 租户对外传播必须包含“租户码 + 租户 Bot 用户名”（可由用户手动输入，不依赖网页）
4) 多副本：每个 Asset 至少 2 份副本分散在不同 Vault Group（避免单群封禁全灭）
5) 副本健康度：记录每个 Vault Group 的成功率/最近失败原因，读路径优先选健康副本
6) 迁移预案（不依赖外部存储的前提下）：
   - 当某 Vault Group 被封且仍存在其他副本：继续服务，不做迁移
   - 当副本冗余不足：由管理员触发“补副本”任务，把现存可用副本再转存到新的 Vault Group 的 Topic（只在 Telegram 内拷贝）

### 6.3 失效场景与处置（Playbook）
#### 6.3.1 Vault Node 被封/不可用
- 症状：copy/forward 到该 chat 失败，或超级群不可访问
- 自动处置：
  - 将节点标记为 degraded/banned，读路径立即绕开该节点
  - 若该节点承载写入，停止写入并改投到其他 active 节点
- 人工处置：
  - 新建 Vault Group 并绑定为 Backup
  - 对“冗余不足”的内容执行补副本（从仍可用副本复制到新 Vault Group 的租户话题）

#### 6.3.2 单个副本消息不可用（message_id 失效等）
- 症状：节点可用，但该 message 无法复制/转发
- 自动处置：
  - 将该 Replica 标记为坏副本，读取切换到其他副本
  - 若冗余不足，进入补副本队列（从可用副本复制生成新副本）

#### 6.3.3 Tenant Bot 被封（租户自有入口失效）
- 目标：缩短恢复时间，尽量保障存量用户可继续访问
- 机制：
  - 预置备用租户 Bot（主/备）共享同一后端与数据库
  - 用户从“应急入口消息”获取备用租户 Bot 用户名
  - 用户在备用租户 Bot 输入租户码或粘贴直达码继续使用

### 6.4 灾备参数建议（MVP 可直接落地）
- 每租户绑定 Vault Group：2 个起（Primary/Backup）
- 单租户 Vault Group 数量：按写入与风险上限配置
- 每个 Asset 副本数 R：2 起（建议 3，分布在不同 Vault Group）
- 最小成功数 minReplicas：2
- 健康检查：每小时抽样 + 关键操作失败触发即时降级
### 6.5 风险边界声明
在不使用域名与外部存储的前提下，无法做到“入口与内容永久不失效”。本项目通过备用租户 Bot 与副本分散，将“单点封禁”风险降低，但无法消除 Telegram 平台级全局封禁风险。

 

## 7. 数据与用户画像

### 7.1 数据分层（不存媒体文件）
- 元数据：Tenant、User、Asset、Replica、Collection、权限规则、Bot 配置、Vault Node
- 行为数据：Event（浏览、打开、搜索、点赞、收藏、推送点击、客服会话等）
- 运营数据：Segment（人群标签）、Broadcast（推送任务与结果）

### 7.2 画像特征（可解释、可运营）
按租户维度生成：
- 内容偏好：标签/类型分布、收藏主题
- 活跃度：近 1/7/30 天打开次数、回访周期
- 意图：搜索关键词簇、近期高频访问目录
- 触达敏感度：推送送达/点击/退订或拉黑信号（不可逆事件）

### 7.3 画像应用（MVP）
- 分群推送：按标签/活跃度分组触达
- 搜索排序：对常看标签提高排序权重（可选）
- 客服侧展示：客服看到用户最近行为与偏好，提升回复效率

## 8. 非功能需求

### 8.1 安全与合规
- 多租户隔离：任何接口与查询必须带 tenant_id 约束
- 最小权限：RBAC 校验默认拒绝
- 审计日志：关键操作（权限变更、删除、补副本、推送）必须记录
- 数据最小化：不采集无必要的个人信息，只记录 Telegram user_id 与必要画像特征

### 8.2 可靠性
- 幂等：上传批次、补副本、推送任务需要幂等键避免重复执行
- 重试：Telegram API 错误分类重试（429 需退避）
- 限流：对推送与批量操作有全局与租户级限流

### 8.3 可观测性
- 结构化日志：带 tenant_id、user_id（可脱敏）、trace_id
- 指标：发送成功率、429 比例、Vault Node 健康度、搜索耗时、推送队列长度
- 告警：Vault Node 连续失败、推送失败率异常、搜索超时

## 9. 交互与命令（建议）
用户侧（租户 Bot）：
- /start：绑定租户（支持输入租户码）
- 搜索：输入关键词或点“搜索”按钮
- 我的：收藏、历史、已加入租户列表
- 客服：在租户上下文内发起

租户侧（租户 Bot 管理台）：
- 租户配置
- 成员管理 / 角色管理
- Vault Nodes 管理（添加频道/群、健康检查）
- 上传批次管理（补充信息、发布、回滚）
- 推送管理（创建、暂停、统计）
- 权限与目录管理

### 9.1 交互分层（一级/二级/三级）
一级交互（全局导航）：
- 使用 ReplyKeyboard（底部菜单）作为租户 Bot 的常驻入口
- 保留 / 命令作为兜底入口（避免菜单被用户收起或误操作）
- 一级菜单建议固定 4~6 项，不超过 2 行：
  - 搜索
  - 我的（收藏/历史/已加入租户）
  - 租户（切换租户/输入租户码）
  - 客服
  - 管理台（仅管理员可见，可选）
  - 设置（可选）

二级交互（上下文列表与选择）：
- 使用 InlineKeyboard（内联按钮）+ callback_data，优先通过编辑同一条消息进行分页与筛选，减少刷屏
- 二级按钮建议统一模板：
  - 翻页：上一页 / 下一页 / 返回
  - 筛选：类型 / 标签 / 时间
  - 排序：相关 / 最新 / 最热
- callback_data 不承载复杂数据，使用短 token 或短 id 映射到后台状态

三级交互（内容卡片与直达）：
- 使用 InlineKeyboard 作为内容动作区：
  - 打开/领取
  - 收藏/取消收藏
  - 点赞/取消点赞
  - 分享（生成直达方式）
- “哈希链接/哈希码直达”用于可转发、可复制的入口：
  - 深度链接直达：t.me/<TenantBot>?start=t_<code>
  - 可复制直达码：T_<code>（用户回到租户 Bot 粘贴即可直达）
- 直达 code 使用短码映射并带版本前缀（例如 t1_<code>），权限校验必须在租户 Bot 内完成

### 9.2 消息呈现与状态策略
- 列表页与筛选页优先“编辑消息”更新内容；仅在关键节点（例如内容交付、客服回投）发送新消息
- 对所有可重复触发的按钮回调做幂等处理（同一 token 重复点击不产生副作用）
- 所有用户状态必须可从后台恢复（不依赖单条消息的局部上下文），避免用户翻旧消息导致状态错乱

## 10. 项目架构（建议，适配纯 Telegram）

### 10.0 立项前置验证（Gate）
必须在目标环境完成以下验证并记录结果，否则不进入大规模开发：
- 话题写入：对 Vault Group 的指定 message_thread_id，验证 sendMessage 成功率与错误码分类
- 话题复制：验证 copyMessage 到指定 message_thread_id（从用户私聊来源与群来源至少各验证一次）
- 用户交付：验证从 Vault Group 的 message_thread_id copyMessage 到用户对话，且用户侧不可见 Vault 来源（符合交付预期）
- 失败分类：至少覆盖 400/401/403/429/5xx 与网络异常，并确认重试/退避策略可行

### 10.1 模块划分（单仓库分层）
- bot/tenant：租户 Bot 交互与路由
- core：领域模型（Tenant、Asset、Replica、Permission、Event）
- services：业务服务（上传编排、检索、推送、客服、画像）
- infra：Telegram API 适配、数据库/缓存/搜索（可选）适配

### 10.1.1 目录与架构资产规范
- arch/components：组件关系图与模块边界图
- arch/flows：关键业务流程图（上传/交付/检索/推送/客服）
- arch/data-model：核心数据模型图与关系说明
- arch/interfaces：对外与跨模块契约清单（Bot ↔ 服务、服务 ↔ 基础设施）
- 命名规则：<领域>_<流程>_<版本>，示例 tenant_upload_v1

### 10.2 关键流程（高层）
- 上传：消息接收 → 归并为批次 → 写入多个 Vault Group 的租户 Topic → 记录 Replica → 生成内容卡片
- 打开内容：用户请求 → 权限校验 → 选择健康副本 → 交付（转发/重新发送）
- 搜索：校验租户与权限 → 执行索引查询 → 返回分页卡片 → 记录搜索事件
- 推送：生成任务 → 分批发送 → 记录结果 → 汇总统计
- 客服：创建工单 → 转发消息 → 回投回复 → 记录会话指标

### 10.2.1 最小闭环落地顺序
- bot/tenant：上传、保存决策与打开请求入口
- services/use-cases：上传编排、权限校验、交付选择
- infra/telegram：copyMessage 与交付能力适配
- core/domain：Asset、Replica、Permission 的最小模型
- infra/persistence：最小元数据落库与幂等

### 10.3 运行时组件（准备开发）
Bot 与服务建议解耦为“交互入口 + API 服务 + Worker”三类进程，避免把长耗时任务卡在 update 处理线程。

- 租户 Bot（入口进程）：
  - 负责：命令与按钮回调、状态机编排、权限校验入口、内容交付、客服回投
  - 不负责：批量补副本、推送大任务、索引重建（全部丢给 Worker）
- API 服务（业务与数据）：
  - 负责：租户/成员/权限、内容元数据、搜索接口、画像与统计、幂等与审计
  - 对外只服务 Bot（未来如需要管理后台 Web，再复用同一 API）
- Worker（后台任务）：
  - 负责：副本写入重试、补副本、Vault Group 能力探测、健康检查、推送队列、索引同步、数据清理
- 数据存储（不存媒体）：
  - 主库：租户/用户/内容元数据与关系（Tenant、Topic 映射、Asset、Replica、Permission、Event）
  - 缓存：短 token、状态机快照、限流计数（可选）
  - 搜索索引：仅存可检索字段（标题/标签/描述/文件名），不存文件（可选）

### 10.4 核心数据模型（概念表）
元数据（强一致）：
- Tenant：租户基本信息、search_mode、默认可见性
- TenantMember：租户成员与角色
- VaultGroup：租户级群配置与状态（active/degraded/banned），能力矩阵、健康度
- TenantVaultBinding：tenantId → vaultGroupChatId 绑定关系（primary/backup/cold）
- TenantTopic：tenantId + vaultGroupChatId → messageThreadId（可版本化）
- Asset：逻辑内容（标题、描述、标签、类型、可见性、searchable）
- AssetReplica：副本定位（vaultGroupChatId、messageThreadId、messageId、file_unique_id、hash、size、状态）
- Collection：目录/合集与继承规则
- PermissionRule：按 Collection/Asset 的 allow/deny 规则

行为数据（可写多、读聚合）：
- Event：浏览/打开/搜索/点赞/收藏/推送点击/客服消息等
- SupportTicket：客服会话与消息映射（forwardedMessageId ↔ userId）
- Broadcast：推送任务、分群条件、发送进度与结果汇总
- AuditLog：运维与敏感操作记录（创建 Topic、迁移切换、补副本、权限变更）

### 10.5 队列与任务（必须项）
为避免 Telegram 429 与长任务阻塞，需要统一任务队列与优先级：
- P0：用户交付（打开内容复制/重试）
- P1：上传写入（副本写入与最小成功数达成）
- P2：推送发送（分批、限流、失败回收）
- P3：补副本/迁移（冗余不足修复、热点租户隔离）
- P4：健康检查/能力探测（巡检与告警）

任务必须支持：
- 幂等键（避免重复写入/重复推送）
- 可重试与退避（429/网络错误）
- 可暂停/可恢复（推送与迁移需要）

### 10.6 Vault Group 配置与租户扩容（平台侧）
- 维护租户级 Vault Group 列表，记录健康度、能力矩阵、风险标签
- 扩容策略（MVP）：Primary 保障写入，Backup 用于容灾与限流分散
- 热点租户扩容：当租户写入/读取量超过阈值，新增 Vault Group 并迁移写入

### 10.7 对外契约（Bot ↔ 服务）
为保证可替换 Bot（租户 Bot 主/备）与可扩展，Bot 不直接拼业务规则，统一调用服务层：
- 权限判断：canUserOpenAsset(userId, tenantId, assetId)
- 副本选择：selectReplica(tenantId, assetId, preferredVaultGroupId?)
- 写入编排：beginUploadBatch / commitBatch / enqueueReplications
- 搜索：searchAssets(tenantId, userId, query, filters, pageToken)
- 行为埋点：trackEvent(event)

### 10.8 部署形态（纯 Telegram 入口）
- 租户 Bot 主/备：多 Bot 身份共享同一后端与数据库，只替换 Bot Token 与限流配置
- Worker 可横向扩展：按任务优先级与租户维度分片，避免单租户占满队列
- 灾备演练：定期模拟“单 Vault Group 不可用”“主租户 Bot 不可用”，验证切换与应急入口消息可达

## 11. 代码风格（适用 TypeScript/Node 方向）
本节定义未来实现时的统一规范；若改用 Python，需要在实现前补充对应规范。

### 11.1 通用原则
- KISS：优先简单可维护方案，避免过度抽象
- DRY：复用以“明确边界的公共模块”为前提，避免到处抽象工具函数
- SRP：一个模块只负责一类职责（交互路由/业务服务/存储适配分离）
- 显式优于隐式：关键参数与权限判断必须显式传递 tenant_id 与 actor

### 11.2 命名与结构
- 文件名：kebab-case（如 user-profile.service.ts）
- 类/类型：PascalCase；函数/变量：camelCase；常量：UPPER_SNAKE_CASE
- 领域对象：以名词命名（Asset、Replica、Tenant）；服务以动词短语命名（buildReplicas、sendBroadcast）

### 11.3 类型与错误处理
- 开启 strict，禁止 any（必要时使用 unknown 并在边界处收敛）
- 返回值使用显式类型；跨模块接口需要定义 DTO/类型
- 错误分层：
  - DomainError：业务可预期错误（无权限、资源不存在、配额不足）
  - InfraError：外部依赖错误（Telegram API、DB、搜索）
- 不吞错：必须记录错误上下文（tenant_id、操作、外部错误码）

### 11.4 日志
- 结构化日志字段建议：level、tenantId、userId（可选）、op、traceId、durationMs、errorCode
- 禁止记录敏感信息（Bot Token、用户隐私内容原文可按需脱敏）

### 11.5 测试（方向）
- 业务服务用单元测试覆盖核心规则（权限、检索策略、分群计算、幂等）
- Telegram 交互层用集成测试验证路由与回调

## 12. 注释风格
默认“少注释、重命名、重结构”，仅在以下场景写注释：
- 规则/算法的“为什么”而非“做什么”
- 与 Telegram 平台限制相关的关键决策（例如 429 退避策略、消息长度限制、转发限制）

规范：
- 注释必须完整句子，说明背景与约束
- 禁止无意义注释（如“set userId”）
- 在公共 API/服务边界使用简短 docstring 说明输入输出与幂等要求

## 13. 里程碑（建议）
- M1：租户 Bot + 多租户 + Vault Group/Topic + 多副本 + 上传/打开闭环
- M2：检索开关 + 搜索 + 收藏/点赞/历史
- M3：客服 + 推送 + 基础画像与分群
- M4：增强检索（OCR/语音转文）与推荐排序（可选）
