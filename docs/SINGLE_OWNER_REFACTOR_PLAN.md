# 单人项目化重构方案

## 1. 目标

本次重构的目标不是推翻现有生产系统，而是在尽量保持以下能力不变的前提下，把项目收敛成“单人维护、单项目心智、权限清晰”的版本：

- 现有生产数据继续可用
- 旧 `shareCode` 尽量继续可访问
- 上传、保存、打开、管理作品的主链路保持可用
- 迭代时尽量不再出现“设置像被重置”的错觉
- 开发和维护不再被多租户、多存储群、多角色治理拖累

本方案默认：

- 不直接在 `main` 上做大改
- 先在独立分支 `codex-simplify-single-owner` 推进
- 先做兼容式简化，再决定是否做破坏性 schema 清理

## 2. 现状判断

### 2.1 当前复杂度来源

项目虽然已经没有真正落地的 Portal Bot，但底层仍然保留了一整套多租户系统设计：

- 配置层以 `PROJECT_CODE`、`PROJECT_NAME` 为主，并兼容 legacy `TENANT_CODE`、`TENANT_NAME`
- 启动时会检查租户一致性，避免写到错误 tenant
- 数据库绝大多数业务表都带 `tenantId`
- 权限依赖 `TenantMember`
- 存储治理依赖 `TenantVaultBinding`、`TenantTopic`
- 读取/交付依赖 `AssetReplica`
- 多个设置以 `TenantSetting(tenantId, key)` 存储

这意味着当前问题不是“有几个多余页面”，而是“多租户抽象已经进入配置、数据库、服务层、Bot 层、worker 层”。

### 2.2 为什么会出现“设置被重置”

从现有实现看，“设置被重置”更像是命中了另一个 tenant，而不是旧设置真的被清空：

- 设置按 `tenantId` 维度存储在 `TenantSetting`
- `delivery-core` 会按 `PROJECT_CODE`（或 legacy `TENANT_CODE`）自动 `upsert tenant`
- 一旦 `PROJECT_CODE`/`TENANT_CODE` 或数据库连接发生漂移，就可能进入一个新的 tenant
- 从用户视角看，结果就像“所有设置恢复默认”

因此，第一优先级不是删代码，而是先把“单 tenant 收口”和“环境稳定性”做好。

### 2.3 为什么旧分享作品会失效

当前分享入口不是单纯的链接问题，而是整条交付链路都可能影响结果：

- `shareCode -> asset`
- `asset -> replica`
- `replica -> vault group / topic / message`
- `worker` 是否已补齐副本
- vault 群或 topic 是否仍有效

也就是说，旧分享失效并不一定是 `shareCode` 失效，更可能是副本链路状态复杂，导致打开时找不到可用交付路径。

## 3. 重构原则

本次重构遵循以下原则：

- KISS：先把系统收敛到单项目心智，再考虑是否删表删字段
- DRY：不要再维持两套“单人项目”和“多租户平台”并行逻辑
- YAGNI：不为未来可能出现的平台化需求继续保留过重抽象
- 单一职责：把“单项目运行能力”和“历史兼容层”分开处理

补充约束：

- 不主动破坏旧 `shareCode`
- 不直接做需要停机的迁移
- 不在未验证前删除旧表
- 不在主分支上边做边试

## 4. 推荐路线

推荐路线不是“从 0 重写”，而是“兼容现网的单人项目化重构”。

### 阶段 A：分析与收口设计

目标：

- 找出多租户、多存储群、多角色相关入口
- 明确哪些能力必须保留，哪些入口应先隐藏
- 形成单人项目运行边界

本阶段产出：

- 本文档
- 文件级改造清单
- 分阶段实施方案

### 阶段 B：单租户运行收口

目标：

- 保留现有数据库结构
- 只允许固定单 tenant 运行
- 将权限心智收敛成“owner / 普通用户 / 访客”
- 隐藏或禁用多 tenant、多管理员、多 backup 群治理入口

建议实施内容：

- 固定唯一 tenant，不再允许生成第二个 tenant
- 将租户相关文案、页面、入口改成“项目设置”或“我的项目”
- 弱化 `TenantMember` 的运营意义，仅保留最低限度兼容
- 将 vault 管理面缩减为“主存储群 + 可选兜底配置”
- 优先保证旧 `shareCode`、旧 `Asset`、旧 `AssetReplica` 继续可用

这是本次重构最关键、最值得先做的一步。

### 阶段 C：服务层简化

目标：

- 让代码心智真正从“多租户平台”变成“单项目系统”
- 把服务接口里的租户概念逐步内隐化

建议实施内容：

- 收敛 `getTenantId()` 的扩散使用
- 将一批“tenant admin”接口改造成“project owner”接口
- 将 settings 读取封装成单项目配置服务
- 重新整理 worker 的职责边界，弱化多副本治理复杂度

### 阶段 D：可选的 schema 清理

只有当阶段 B、C 已验证稳定时，才考虑执行本阶段。

候选清理对象：

- `TenantMember`
- `TenantVaultBinding`
- `TenantTopic`
- `PermissionRule`
- 业务表中的显式 `tenantId` 依赖

本阶段风险最高，因为一旦删表删字段，就会影响回滚能力。没有明确收益前，不应急于推进。

## 5. 为什么不建议直接从 0 重写

如果目标是：

- 继续使用现有生产数据
- 尽量保留旧分享入口
- 不中断已有用户使用

那么从 0 重写未必更省。真正昂贵的部分不是写代码本身，而是：

- 行为重新对齐
- 数据兼容
- 旧链接兼容
- worker 行为兼容
- 生产切换验证

只有在明确愿意砍掉大量现有能力时，纯重写才可能更便宜。

## 6. 工程量评估

### 方案一：只做兼容式简化

范围：

- 不改 schema
- 不删表
- 只做入口收口、权限收口、运行时单 tenant 化

预估：

- 3 到 5 天可完成第一版

风险：

- 低到中

收益：

- 快速降低维护复杂度
- 对生产最友好

### 方案二：兼容式简化 + 服务层重构

范围：

- 收口运行边界
- 重整 service / worker 结构
- 保持数据兼容

预估：

- 1 到 2 周

风险：

- 中

收益：

- 真正把项目改造成“单人项目心智”

### 方案三：彻底删除多租户结构

范围：

- Prisma schema 调整
- 数据迁移
- 历史兼容层清理

预估：

- 2 到 4 周

风险：

- 高

收益：

- 长期结构最干净

不建议直接从这个阶段开始。

## 7. 第一阶段实施边界

后续自动任务和手动改造都应遵守以下边界：

- 只在 `codex-simplify-single-owner` 分支工作
- 不合并 `main`
- 不回滚用户已有未提交改动
- 不直接改生产环境
- 不直接做破坏性数据库迁移
- 每轮只完成一个小阶段
- 每轮改动后至少尝试运行 `npm run test` 和 `npm run build`
- 发现高风险决策时先在当前线程汇报

## 8. 下一步建议

建议下一阶段直接进入“阶段 B：单租户运行收口”，但先只做以下低风险改造：

- 明确需要隐藏的多租户/多管理员/多存储群入口
- 收敛文案与页面命名
- 固定单 tenant 运行边界
- 梳理旧 `shareCode` 兼容测试点

在这一步完成前，不建议动 Prisma schema。
